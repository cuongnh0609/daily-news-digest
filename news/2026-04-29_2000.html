B1Claude Code サブエージェントに潜む32K/64Kトークン出力上限 — 原因と4つのワークアラウンド

📰 本文 + 和越対訳

Claude Code 上で stdio 形式の MCP サーバーを自作する際、コードを修正してビルドし直した後に /mcp reconnect しても変更が反映されないという落とし穴について、フロントエンドエンジニアがZennで詳細な解説記事を公開した。原因はNode.jsのプロセスライフサイクルにある。.mcp.json の command に指定されたコマンドは、Claude Code 起動時に子プロセスとしてspawnされ、dist/index.js をその時点でメモリに読み込む。以降、ディスク上のファイルが更新されても実行中のプロセスには反映されない。reconnect はパイプの再確立とMCPハンドシェイクのやり直しにすぎず、プロセスそのものを入れ替えるわけではない。解決策は古いプロセスをkillして reconnect すること（ps aux | grep <サーバー名> でPIDを特定 → kill <PID> → /mcp reconnect）。Claude Code がプロセスのライフサイクルを自動管理していない点は公式issueでも多数報告されており（#3095, #7174, #22612）、現在も未解決だ。開発効率のためにシェルエイリアスで半自動化するのが現実的な対処法とされる。

Một kỹ sư frontend Nhật Bản đã đăng bài phân tích chi tiết trên Zenn về vấn đề phổ biến khi tự xây dựng MCP server theo kiểu stdio cho Claude Code: sau khi sửa code và build lại, lệnh `/mcp reconnect` không phản ánh thay đổi. Nguyên nhân nằm ở vòng đời tiến trình Node.js — MCP server được spawn như một child process khi Claude Code khởi động và load file `dist/index.js` vào bộ nhớ lúc đó. Dù file trên đĩa được cập nhật, tiến trình đang chạy không tự nạp lại. Lệnh `reconnect` chỉ thiết lập lại kết nối pipe, không thay thế tiến trình. Cách giải quyết: dùng `ps aux | grep ` để tìm PID → `kill ` → `/mcp reconnect` để spawn process mới. Tình trạng Claude Code không tự quản lý vòng đời tiến trình MCP đã được báo cáo trong nhiều issue (chưa được giải quyết). Tác giả gợi ý dùng shell alias để bán tự động hóa bước kill-reconnect.

C1Bugs Rust Won't Catch: 8 Security Pitfalls Found in Rust's coreutils Reimplementation — 44 CVEs, Zero Memory Safety Flaws

📰 Article (English)

When Canonical disclosed 44 CVEs in uutils — the Rust reimplementation of GNU coreutils shipping by default since Ubuntu 25.10 — a striking pattern emerged: none were memory safety bugs. No buffer overflows, no use-after-free, no null-pointer dereferences. The borrow checker did its job perfectly. But 44 vulnerabilities still made it through. Matthias Endler from corrode Rust Consulting analyzed the bugs and distilled eight systematic failure categories that define the new security boundary in Rust programs. The largest cluster was TOCTOU (Time-Of-Check-Time-Of-Use): trusting a path across two consecutive syscalls. Rust's fs::metadata, File::create, and fs::remove_file all re-resolve paths independently — creating windows for symlink swapping attacks. The fix is anchoring on file descriptors using OpenOptions::create_new(). Other categories include setting permissions after file creation (use .mode() at creation instead), comparing paths as strings rather than by inode (always canonicalize()), and treating String/&str as if they were bytes at Unix system boundaries — when paths, env vars, and streams are actually raw bytes requiring OsStr or &[u8]. The worst single bug was a local root code execution in the chroot implementation: NSS shared libraries were dynamically loaded (dlopen) after the chroot call, executing attacker-controlled code from the new root filesystem. The fix: resolve all user lookups before crossing the trust boundary. A key takeaway is that the new security boundary for Rust programs is not memory — it is the interface between Rust and the OS: paths, bytes, and syscalls.

This release continues LLM's philosophy of being the fastest path to accessing newly released models from the terminal. Beyond GPT-5.5 support, LLM 0.31 introduces per-invocation verbosity controls that let users suppress or expand model output metadata without modifying global configuration — a long-requested quality-of-life feature for scripting workflows. Image detail options are also new: users can now specify low, high, or auto detail levels when passing images to vision-capable models, directly controlling the token cost and quality tradeoff of image analysis. The release also includes improvements to the plugin ecosystem: several community plugins for Anthropic Claude models, Google Gemini, and local inference engines like Ollama received compatibility updates. LLM's architecture — where each model provider is a separate installable plugin — means the tool consistently tracks the latest model releases across all major providers within days of announcement. LLM is distributed as a Python package via pip and is positioned as the Swiss Army knife for LLM power users, researchers, and developers who prefer terminal-first workflows over chat UIs.
