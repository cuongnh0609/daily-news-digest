B1Claude Codeに「昨日の続き」を覚えさせるCLI「lk」——SQLite FTS5とGitで実装するAIエージェント向けローカル知識ベース

📰 本文 + 和越対訳

AndroidエンジニアのSyarihuが公開したRust製CLI「lk」は、Claude Codeのセッション間にわたる記憶の欠如という根本的な問題に取り組むツールだ。Claude Codeは同一プロジェクトの次のセッションでも、前回の調査結果や設計判断を知らない状態で始まる。lkはこれを解決するため、2層構造の知識ベースを提供する。ローカル層はSQLiteデータベースで一時的な調査キャッシュを管理し、共有層はGitで追跡するMarkdownファイルに安定した設計決定・チーム規約を保存する。検索は3段階の戦略を採用しており、FTS5トリグラムによる全文検索、キーワード補完、そして短いCJK文字列向けのLIKEフォールバックを組み合わせることで、精度よりもリコールを優先する。古くなったエントリは削除せず「期限切れ」フラグを立てるため、Claudeが現在のコードと照合して有効性を確認できる。CLAUDE.mdにlkを参照する指示を追加するだけで、ClaudeはコードベースのUnknown部分を調査する前に自動的に知識ベースを検索するようになる。プロジェクト単位とユーザーグローバル単位の2スコープをサポートし、ヘキサデシマルIDでエントリ衝突を防ぐ設計が特徴だ。

"lk" là một CLI viết bằng Rust, giải quyết vấn đề căn bản của Claude Code: mỗi phiên làm việc mới đều bắt đầu từ đầu, không nhớ kết quả điều tra hay quyết định thiết kế từ hôm qua. Công cụ cung cấp kiến trúc 2 tầng: tầng local dùng SQLite quản lý cache điều tra tạm thời, tầng shared dùng Markdown file theo dõi bằng Git lưu các quyết định kiến trúc ổn định và quy ước nhóm. 📰 Article (English)

Systima conducted a rigorous benchmark measuring actual API token consumption between Claude Code and OpenCode by intercepting traffic at the proxy boundary between harnesses and the model endpoint. The results reveal stark differences in baseline efficiency. Claude Code consumes approximately 33,000 tokens of system prompts and tool schemas before the model even sees user input, compared to OpenCode's ~7,000 tokens — a 4.7× difference on simple tasks. This gap compounds dramatically with real-world configuration: a 72KB instruction file adds ~20,000 tokens per request for both systems, and five MCP servers contribute another 5,000–7,000 tokens. Cache behavior proved the most striking divergence. OpenCode maintains byte-identical prefixes throughout sessions, enabling optimal cache hits. Claude Code produces three distinct request classes per session with different cache prefixes, leading to mid-session cache rewrites. On identical tasks, Claude Code wrote 5.9× to 54× more cache write tokens than OpenCode. Subagent delegation carries a severe multiplier: a task costing 121,000 tokens directly ballooned to 513,000 tokens when distributed across two subagents — a 4.2× explosion. However, on multi-step tasks, Claude Code's aggressive tool-call batching partially offsets its higher baseline. The paper emphasizes measuring actual API traffic rather than estimates, and provides a reproducible methodology using a transparent proxy for production AI systems auditing.

📰 Article (English)

Anthropic has published detailed technical documentation on Fable 5's cybersecurity safeguards alongside an early-draft Cyber Jailbreak Severity (CJS) scale, designed to standardize how AI security breaches are assessed and communicated across the industry. The CJS scale grades jailbreaks from CJS-0 (informational) to CJS-4 (critical) based on four axes: capability gain (how much the jailbreak advances attacker capability beyond existing tools), breadth of capability gain (cross-domain applicability), ease of weaponization (required technical expertise), and discoverability (whether the jailbreak is publicly known). Tier 2 (High-risk dual use) blocks penetration testing and exploit development pending better access controls. Tier 3 (Low-risk dual use) allows vulnerability identification with defensive framing, partially blocked within a "safety margin." Tier 4 (Benign use) permits secure coding, patch management, and threat analysis. Anthropic explicitly acknowledges cybersecurity's dual-use tension: capabilities benefiting defenders can also empower attackers. The framework accepts higher false-positive rates on Tier 3 to ensure margin for error, explicitly trading off some legitimate security research capability for harm prevention. This CJS scale aims to bring CVSS-like rigor to AI jailbreak reporting, enabling consistent severity assessment across different AI systems.

Đồng thời, tài trợ liên bang Mỹ cho khoa học toán học bị cắt giảm đáng kể, làm yếu đi pipeline con người cần thiết để hiểu, kiểm chứng và thách thức các kết quả AI đó. Park định vị năng lực toán học — "khả năng được rèn luyện để kiểm chứng, diễn giải và thách thức lý luận toán học" — như cơ sở hạ tầng thiết yếu, tương tự ngành sản xuất chip bán dẫn. Không như chip, năng lực này được xây dựng qua hàng thập kỷ thông qua các cơ cấu học thuật và không thể tái tạo nhanh nếu bị phá vỡ. Giải pháp đề xuất: yêu cầu AI thực hiện lý luận toán học hệ quả phải công khai các luận điểm quan trọng ở định dạng có thể kiểm chứng bằng máy (formal proofs), biến lý luận AI từ "thuyết phục mờ đục thành cấu trúc có thể kiểm toán". Lập luận này áp dụng trực tiếp cho kỹ thuật phần mềm: khi AI tạo ra code, quyết định kiến trúc và phân tích bảo mật, những kỹ sư có thể thực sự audit output đó — không chỉ chạy test — trở thành bottleneck quan trọng.
