06AIエージェントが毎回データを取りに行く設計の限界——Memory-First アーキテクチャへの転換

📰 本文 + 和越対訳

AIエージェント開発において広く採用されてきた「scatter-gather」パターン——つまりエージェントがタスクのたびに複数の外部システムへ問い合わせてデータを収集する設計——が、本番環境での採用を阻む本質的な限界を抱えていると指摘する記事がZennに公開された。著者は、このアプローチが3つのコスト構造上の問題を引き起こすと論じている。第一に、最も遅いAPIが全体のレイテンシを決定してしまうこと。第二に、毎回コンテキストをゼロから再構築するためトークン消費量が膨大になること。そして第三に、CRM・HR・サポート等の各システムで「顧客」の呼称が異なるため、システム間の関係性が失われた断片的な回答しか生成できないこと。代替として提唱されるのが「Memory-first」設計であり、ナレッジグラフを事前統合済みの記憶層として機能させ、エージェントは推論前に必要な関係情報にアクセスできる状態にしておくアプローチだ。

Một bài viết trên Zenn chỉ ra giới hạn cốt lõi của mẫu thiết kế "scatter-gather" - nơi AI agent mỗi lần thực hiện task đều đi truy vấn nhiều hệ thống ngoài để thu thập dữ liệu - khiến việc triển khai thực tế gặp rào cản. 10npm Staged Publishing & Install-Time Source Controls: Human-Gated Releases for Supply Chain Safety

📰 Article (English)

GitHub has shipped two supply chain security features for npm in version 11.15.0. The first, Staged Publishing (now generally available), requires a maintainer with two-factor authentication to explicitly approve a package before it becomes installable. When a CI/CD pipeline publishes using trusted publishing (OIDC), the tarball enters a queue visible on npmjs.com and the npm CLI — but no one can install it until a human with 2FA hits approve. This applies even to fully automated pipelines, effectively making every npm release a two-person operation. The second feature adds three new install-time flags: --allow-file (controls installs from local file paths and tarballs), --allow-remote (controls installs from remote URLs), and --allow-directory (controls installs from local directories). Each flag accepts all (current default behavior) or none. These can be locked in .npmrc or package.json, enabling teams to enforce strict allowlists for where their dependencies originate. Combined with the existing --allow-git flag, teams now have granular control over every dependency source type — significantly narrowing the attack surface for compromised or malicious packages.

11Constraint Decay: LLM Agents Lose ~30 Points as Backend Structural Requirements Accumulate

📰 Article (English)

A new research paper (arXiv: 2605.06445) introduces the concept of "constraint decay" — a systematic performance collapse observed when LLM coding agents face increasing structural requirements in backend code generation. While agents perform well on loosely specified tasks, production backend software demands adherence to architectural patterns, ORM conventions, database schemas, and framework idioms simultaneously. The researchers evaluated agents across 80 greenfield and 20 feature-implementation tasks spanning eight web frameworks (including Flask, FastAPI, and Django) using both behavioral tests and static code verifiers. Their headline finding: capable agent configurations lose approximately 30 percentage points in assertion pass rates when moving from minimal to fully specified structural requirements. Some weaker configurations approach zero performance on complex tasks. The root cause identified is "data-layer defects" — specifically incorrect query composition and ORM runtime violations — as agents struggle to simultaneously satisfy both functional correctness and structural compliance. Flask-based tasks performed best due to minimal conventions, while FastAPI and Django tasks showed dramatic degradation. The paper frames jointly satisfying functional and structural requirements as an unsolved problem, highlighting a critical gap between benchmark performance (often measured on loose tasks) and real-world backend engineering.
