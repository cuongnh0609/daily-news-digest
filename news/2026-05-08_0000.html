C-01Unsloth + NVIDIA Collaboration Achieves 25% Additional LLM Fine-Tuning Speedup via Three Kernel-Level Optimizations

📰 Article (English)

Unsloth and NVIDIA have published a joint technical post detailing how three targeted optimizations in the "glue code" surrounding GPU kernels deliver roughly 25% additional throughput for LLM fine-tuning — on top of Unsloth's existing 2–5× speedups over baseline PyTorch training. The first technique, Packed-Sequence Metadata Caching, concatenates short training examples into single packed sequences and caches sequence boundary metadata (lengths, offsets, attention structures) across all transformer layers instead of recomputing it at every layer. On Qwen3-14B QLoRA supervised fine-tuning, forward pass improved by 43.3% and backward pass by 5.8%, yielding +14.3% per batch. The second technique, Double-Buffered Checkpoint Reloading, pipelines activation checkpointing by using two CPU pinned-memory buffers so the GPU can compute backward on one activation while preloading the next — hiding data-movement latency. NVIDIA Blackwell B200 benchmarks show +8.4% for 8B models, +6.7% for 14B, +4.6% for 32B. The third, MoE Routing Optimization, replaces repeated dynamic expert-assignment queries with a single grouped assignment, stable sort, and bincount — reducing synchronization overhead to yield +10–15% on mixture-of-experts models. All improvements apply automatically across RTX laptops, datacenter GPUs, and DGX Spark systems without any user-side changes.

C-02CVE-2026-31431: Cloudflare Mitigates Critical Linux Kernel "Copy Fail" Vulnerability Using eBPF — No Signature Updates Needed

📰 Article (English)

Cloudflare published a detailed technical post-mortem on CVE-2026-31431, a critical Linux kernel vulnerability dubbed "Copy Fail," and how they mitigated it across their entire fleet before a kernel patch was widely available — using only eBPF tooling, with zero customer impact. The vulnerability is an out-of-bounds write in the Linux kernel's algif_aead module (the authencesn AEAD wrapper). The exploit chain uses splice() combined with page cache manipulation and recvmsg() to write 4 bytes past the intended output buffer boundary, tainting cached file contents. The canonical attack target is /usr/bin/su (a setuid-root binary): injecting shellcode there gives root execution when any user runs su. Cloudflare's response: first, they deployed fleet-wide eBPF tracing to map all legitimate AF_ALG socket users across their infrastructure — finding only one internal service actually needed the interface. Second, they wrote a BPF-LSM security hook that blocks socket_bind for the AF_ALG family for all processes except that one allowlisted service. Third, the kernel patch was rolled out via existing automated reboot infrastructure. During authorized red-team testing, their behavioral detection flagged the exploit attempt within minutes — without any signature or rule changes.

📰 Article (English)

Val Town's engineering team published a candid post-mortem on their migration journey across three authentication platforms in 18 months — from Supabase Auth, to Clerk, and finally to Better Auth — providing a detailed technical breakdown of why each failed and what the team now recommends. Supabase Auth was adequate initially but became limiting as the platform grew. Clerk was adopted for its polished developer experience, but hit a critical limitation: a hard 5 requests/second rate limit on user-data read endpoints, which made social feed features impossible to serve reliably. Beyond rate limits, Clerk's session management was a single point of failure — Clerk outages caused site-wide unavailability even for already-authenticated users. Better Auth, an open-source framework, solved all three problems: sessions and user data live in Val Town's own database (PostgreSQL), eliminating vendor lock-in and rate limits; optional stateless paid infrastructure is available for scale; and high-quality framework integrations removed the boilerplate. Val Town ran a two-week parallel migration supporting both auth systems simultaneously to ensure zero-downtime transition. The key lesson: for platforms where user data reads are on the hot path (social features, real-time feeds), auth-as-a-service providers that store state remotely become a bottleneck and single point of failure.
