B-1Claude Codeのセッションログから過去のやり取りをたどる方法――JSONL解析で根本原因究明

📰 本文 + 和越対訳

Claude Codeはすべての会話履歴・ツール実行・ファイル編集をJSONL形式でローカルに保存しており、後からいつでも「なぜこの変更が入ったのか」を検証できる。ログは ~/.claude/projects/<プロジェクトパス>/*.jsonl に格納されており、対象セッションの特定には「時間範囲でのファイル絞り込み → キーワードマッチ数ランキング」という3段階アプローチが有効だ。JSONL解析では type フィールドで発話者を判定し、content フィールドの型（配列か文字列か）に注意しながらイベント種別を分類する。記事の著者は「Issueに記載していない制約をAIが独断で追加した」という事例を発端に、このログ分析手法を確立した。単なる事後確認だけでなく、根本原因の特定と再発防止策の立案に一次資料として活用できる点が強調されている。Windows環境ではPowerShellでUTF-8を明示指定することで文字化けを防止できる。

Their toolkit uses a feature.yaml format that assigns unique identifiers (ACIDs) to each acceptance criterion, creating a tight coupling between requirements and code. This allows developers to trace which acceptance criteria are satisfied where in the codebase, enabling AI agents to identify and fix failures autonomously. The approach is essentially a return to rigorous, documented software engineering practices — refined for an era where specs feed directly into agentic workflows. Teams adopting this model shift from prompt engineering to specification engineering, reducing ambiguity that AI agents amplify into unexpected behavior.
