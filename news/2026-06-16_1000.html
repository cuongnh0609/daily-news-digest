The 1.0 release signals API stability, meaning libraries and tools built on iroh today won't break with future updates. For engineers building distributed systems, LLM inference clusters, or multi-agent communication layers, iroh provides a production-ready transport that requires no coordination infrastructure.

C3How Memory Safety CVEs Differ Between Rust and C/C++: Why Comparing Counts Is Misleading

📰 Article (English)

A detailed technical post from Jakub Beránek (a Rust compiler contributor) explains why comparing the raw count of CVEs between Rust and C/C++ software is misleading and often used in bad faith. The key insight is structural: in C and C++, memory safety bugs are exploitable vulnerabilities because the language provides no guarantees about memory access — a buffer overflow or use-after-free silently corrupts state and can be weaponized. In Rust, most memory safety issues are surfaced as compile-time errors that never become CVEs at all. The CVEs that do appear in Rust codebases typically fall into one of three categories: logic errors unrelated to memory safety (which any language can have), soundness bugs in the Rust standard library or compiler itself (rare and patched quickly), and issues in unsafe blocks where Rust's guarantees are explicitly suspended. The post provides a methodological framework for how to properly compare security postures across languages, including how to weight the severity and exploitability of CVEs rather than just counting them. For developers evaluating Rust adoption or defending the choice to stakeholders, this piece provides concrete, citable arguments backed by CVE data and Rust's memory model.
