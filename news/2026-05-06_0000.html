Bài viết của ikenyal (GENDA) tiếp cận vấn đề từ góc độ agent tự động hóa nghiệp vụ, không phải agent coding. C2Async Rust Never Left the MVP State — Binary Bloat from State Machine Design Still Needs Compiler Fixes

📰 Article (English)

Dion (Tweede Golf) argues that async Rust's "zero-cost abstractions" promise was never fully delivered — particularly on embedded systems where every byte counts. The core problem is the compiler-generated state machine for each async function: it creates multiple states even for trivial futures (a function that just returns 5 still gets a full Resumed/Returned/Panicked/Suspend state machine), adds expensive panic paths for the Returned state, and generates 360 lines of MIR for a simple two-await function versus 23 for the sync equivalent. Dion proposes four concrete compiler-level optimizations: (1) replace the Returned-state panic with Poll::Pending in release builds, saving 2–5% binary size on embedded firmware; (2) eliminate state machines entirely for async blocks with no await points; (3) implement future inlining for single-await callers; (4) collapse identical states generated by match branches. LLVM cannot compensate for these because panics introduce side effects the optimizer can't eliminate, and futures nested deeply in idiomatic async code exceed LLVM's inlining capacity even at opt-level=3. The author has submitted a Project Goal and is seeking €30k in funding to implement these changes in rustc.
