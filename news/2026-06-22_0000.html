B1Rustの公式MCP SDK「rmcp」でRemote MCPサーバーを実装する——Streamable HTTP transportの実践ガイド

📰 本文 + 和越対訳

Model Context Protocol（MCP）の公式Rust SDK「rmcp」を使ってリモートMCPサーバーを実装する方法が、Zennで詳細に解説された。rmcpはサーバー・クライアント双方をサポートしており、transportとしてstdioとStreamable HTTPを選択できる。Remote MCP Server（HTTPでホストするMCPサーバー）を構築するには、Cargo.tomlでtransport-streamable-http-serverフィーチャーを有効にし、axumと組み合わせてルーターを構築する流れになる。ツールの定義には#[tool]マクロとServerHandlerトレイトを実装し、JSON Schemaの自動生成にはschemarsクレートを活用する。著者は実装例として日本の山岳データAPIを公開しており、MCPエコシステムへの参入コストを大幅に下げる実践的な内容として注目されている。MCPサーバーをRustで書く選択肢が広がることで、パフォーマンスと型安全性を重視するエンジニアに新たな道が開けた。

Một bài hướng dẫn chi tiết về cách implement Remote MCP Server sử dụng "rmcp" — SDK Rust chính thức cho Model Context Protocol (MCP) — đã được đăng trên Zenn. rmcp hỗ trợ cả server và client, cho phép chọn transport là stdio hoặc Streamable HTTP. Để xây dựng Remote MCP Server (MCP server được host qua HTTP), cần bật feature transport-streamable-http-server trong Cargo.toml và kết hợp với axum để xây dựng router. Định nghĩa tool sử dụng macro #[tool] và trait ServerHandler, còn tự động sinh JSON Schema thì dùng crate schemars. Tác giả đã công bố ví dụ implement là API dữ liệu núi Nhật Bản, được đánh giá cao vì giảm đáng kể chi phí gia nhập hệ sinh thái MCP. Việc có thêm lựa chọn viết MCP server bằng Rust mở ra con đường mới cho các kỹ sư coi trọng performance và type safety.

📰 Article (English)

A detailed technical exploration published on sibexi.co compares epoll and io_uring — the two dominant async I/O interfaces available in the Linux kernel — through the lens of building a production-grade reverse proxy. The author, a university lecturer, originally built "TinyGate" as an educational worker-based reverse proxy and discovered its architectural limits under real load. Researching how tools like nginx and haproxy achieve their performance led to deep experimentation with both interfaces. Epoll, the traditional event-notification API in Linux since kernel 2.5.44, works by monitoring file descriptors for I/O readiness and notifying the application when operations can proceed without blocking. io_uring, introduced in kernel 5.1 (2019), takes a fundamentally different approach: it uses a ring buffer shared between kernel and userspace to submit and complete I/O operations asynchronously without syscall overhead per operation. The article benchmarks both interfaces for connection handling, request routing, and upstream proxy operations. Key findings: io_uring achieves significantly higher throughput at high concurrency due to reduced syscall count and batching, but adds implementation complexity. Epoll remains sufficient for moderate-traffic use cases and is more portable across older kernels. The piece also covers how Tokio (Rust async runtime) and other modern runtimes are gradually migrating to io_uring backends.
