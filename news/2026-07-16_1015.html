C-03Designing APIs for Agents: Four Principles That Make or Break Agent-Driven Integrations

📰 Article (English)

A post from the Freestyle engineering blog (surfaced on Hacker News) makes a compelling case that API design for AI agents requires fundamentally different principles than design for human developers. The core argument: since agents can consume entire documentation sets instantly and generate thousands of lines of code without fatigue, human-centric conveniences like smart defaults and forgiving error handling actually become liabilities. The author proposes four principles for agent-first API design: (1) Eliminate defaults — agents should explicitly specify every parameter; implicit defaults create silent inconsistencies across agent-generated code at scale. (2) Embrace precise errors — strict, informative error messages teach agents the exact contract, while forgiving APIs accepting multiple boolean formats generate inconsistent code. (3) Use specific field names — generic names like "name" cause hallucination risks; prefer displayName, slug, externalId. (4) Provide core facts only — avoid utility wrappers; expose raw primitives and let agents build their own abstractions. The team applied these to Freestyle VMs, shifting from SDK abstractions to minimal exec-based APIs with dramatically better agent onboarding.
