C1CodeQL 2.26.0 Adds AI Prompt Injection Detection for JavaScript and TypeScript

📰 Article (English)

GitHub has released CodeQL 2.26.0, introducing a significant new security capability: a JavaScript and TypeScript query called js/system-prompt-injection that detects when untrusted, user-provided values flow into an AI model's system prompt. This type of vulnerability — where attackers manipulate AI behavior by injecting malicious content through unvalidated user input — has become one of the most critical attack vectors in AI-powered applications. The update also expands detection coverage across popular AI platforms, adding support for additional OpenAI APIs (including Sora prompts and Realtime session instructions), Anthropic client APIs, and Google GenAI. Alongside the AI-specific changes, CodeQL 2.26.0 adds Kotlin 2.4.0 support, which is notable for Android and multiplatform developers. The prompt injection query is now integrated into GitHub's code scanning feature and runs automatically on pull requests for repositories using GitHub Advanced Security or the free public repository tier. This means developers can now catch prompt injection vulnerabilities before they reach production, a capability that did not exist in any widely-adopted static analysis toolchain until now.

C3Why Write Code in 2026 — The Case for Human Authorship in the Age of AI Coding Agents

📰 Article (English)

Doug Turnbull makes a direct and practical argument: even as AI coding agents become powerful enough to handle large portions of a codebase, human engineers should continue writing code themselves. His core claim is that writing code is a thinking tool — it forces you to engage with the execution environment in a way that generates understanding that no amount of reviewing AI-generated output can replicate. The act of typing forces precision; natural language is inherently ambiguous, but code is not. Turnbull argues that hands-on coding lets developers recognize architectural fragility before it calculates itself into production, catch testing gaps, and maintain genuine system ownership. He introduces a key analogy: AI coding agents are not like compilers that reliably translate intent into output — they are more like "freshly onboarded interns," capable of doing good work but amplifying both good and bad decisions from the human directing them. If the human's mental model is wrong, the intern's output will be wrong too. The article does not argue against using AI agents. Rather, it argues that engineers who stop writing code will progressively lose the precise mental model needed to direct agents effectively, leading to quality degradation and accumulated technical debt that accumulates quietly in the background.

Doug Turnbull đưa ra lập luận thực tế và trực tiếp: dù AI coding agent ngày càng mạnh đến mức có thể xử lý phần lớn codebase, kỹ sư vẫn nên tự viết code. Luận điểm cốt lõi của ông là: viết code là một công cụ tư duy — nó buộc bạn phải tương tác trực tiếp với môi trường thực thi theo cách tạo ra sự hiểu biết sâu sắc mà chỉ review output của AI không thể thay thế được. Hành động gõ code đòi hỏi sự chính xác; ngôn ngữ tự nhiên vốn mơ hồ, còn code thì không. Turnbull lập luận rằng việc tự viết code giúp developer nhận ra điểm yếu kiến trúc trước khi nó gây hại trong production, phát hiện khoảng trống trong testing, và duy trì quyền sở hữu hệ thống thực sự. Ông đưa ra ẩn dụ then chốt: AI coding agent không giống compiler — chúng giống "nhân viên mới vào việc," có thể làm việc tốt nhưng sẽ khuếch đại cả quyết định đúng lẫn sai của người hướng dẫn. Nếu mental model của con người sai, output của AI cũng sẽ sai. Bài viết không phản đối dùng AI agent, mà lập luận rằng kỹ sư nào ngừng viết code sẽ dần mất đi mental model chính xác cần thiết để điều phối agent hiệu quả.
Bài viết này liên quan trực tiếp đến cách bạn phân chia công việc với Claude Code hoặc Cursor trong dự án hàng ngày. Giao phó toàn bộ implementation mà không tự viết bất kỳ phần nào có thể khiến khả năng debug và thiết kế kiến trúc của bạn suy giảm theo thời gian.
