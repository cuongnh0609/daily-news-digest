B-01AIの請求がサーバ代を超えた日——LLMコスト爆発の三重構造と防ぐための設計原則

📰 本文 + 和越対訳

あるSaaSプロダクトで、LLM APIの利用料金が一日にして一ヶ月分のサーバ代を超えるという事態が発生した。調査の結果、原因は三つの要因が重なったことだった。第一に、新しいデータベースカラムを参照するコードが、そのカラムを追加するマイグレーションより先にデプロイされた。第二に、このエラーをタスクキューが自動でリトライし続けた。第三に、バッチ処理が冪等性（べきとうせい）を持たず、毎回全量のLLM APIを呼び出していた。重要なのはLLM側のAPIは全てHTTP 200で成功しており、課金が発生しつづけていた点だ。著者は「確定的な失敗はリトライで解決できない」こと、「コストの大きい処理は冪等性が命」であることを強調し、環境ごとのAPIキー分離と費用アラートの重要性を訴えている。

Một sản phẩm SaaS đã trải qua tình huống chi phí API LLM trong một ngày vượt quá toàn bộ tiền thuê server một tháng. Điều tra cho thấy ba yếu tố cộng hưởng: (1) code tham chiếu cột database mới được deploy trước khi migration tạo cột đó chạy; (2) task queue tự động retry liên tục khi gặp lỗi; (3) batch process thiếu idempotency nên mỗi lần retry đều gọi toàn bộ LLM API từ đầu. Điểm then chốt là phía LLM API luôn trả về HTTP 200 — tức là đã tính phí đầy đủ — nhưng kết quả bị bỏ qua vì lỗi database sau đó. Tác giả nhấn mạnh: lỗi xác định không thể giải quyết bằng retry, và idempotency là bắt buộc với mọi tác vụ tốn kém.

MCPサーバーを開発環境からクラウドの本番環境へ移行する際には、三つのアーキテクチャ上の課題がある。第一はトランスポート層の刷新で、従来のHTTP+SSEを「Streamable HTTP」に移行することで、単一エンドポイントが必要な場合のみSSEを使う設計に変わる。第二はステートレスなセッション管理で、SDKのstateless=Trueを設定することでセッションIDへの依存を排除し、ロードバランサー配下でも水平スケールが可能になる。ユーザーのコンテキストはOAuth 2.1のベアラートークンで管理し、プロセスローカルなメモリに依存しない。第三は.well-knownエンドポイントによるケイパビリティ発見で、ツール定義や認証要件をJSON形式で公開することで、接続前にクライアントがサーバーの仕様を把握できるようになる。これらを組み合わせることで、真のクラウドネイティブなMCP運用が実現する。

Khi chuyển MCP server từ môi trường dev lên production trên cloud, có ba thách thức kiến trúc cần giải quyết. Thứ nhất là chuyển transport sang "Streamable HTTP" — một endpoint duy nhất, chỉ dùng SSE khi cần phản hồi nhiều message. Thứ hai là kiến trúc stateless session: đặt stateless=True trong SDK để loại bỏ phụ thuộc vào Session ID, cho phép horizontal scale sau load balancer — context người dùng được truyền qua OAuth 2.1 bearer token. Thứ ba là capability discovery qua endpoint .well-known, publish định nghĩa tool và yêu cầu xác thực dạng JSON để client biết trước mà không cần kết nối đầy đủ. Kết hợp ba điểm này tạo ra MCP server thực sự cloud-native.

B-03Claude Codeの!コマンドがシェル出力に自動応答するようになった——デフォルト変更とコストへの影響

📰 本文 + 和越対訳

Claude Code v2.1.186（2026年6月中旬）より、!コマンドで実行したシェルの出力に対してClaudeが自動的に応答を生成するようになった。従来はシェル出力がコンテキストに追記されるだけで、ユーザーが明示的にプロンプトを送る必要があったが、新しいデフォルト動作ではコマンド実行後に自動でClaudeが応答する。この自動応答は通常のプロンプト送信と同等のトークンコストがかかるため、頻繁に!コマンドを使うユーザーにとっては予期しないコスト増になり得る。元の動作に戻したい場合はsettings.jsonに"respondToBashCommands": falseを追加すればよい。著者は、テストやビルドの結果をすぐに解析させたいユーザーには有用だが、状態確認程度の用途にはfalseに設定することを推奨している。

Kể từ Claude Code phiên bản v2.1.186 (giữa tháng 6 năm 2026), lệnh ! dùng để chạy shell command giờ tự động kích hoạt Claude phản hồi output — không cần người dùng gửi thêm prompt. Trước đây, output chỉ được append vào context và người dùng phải hỏi thêm thủ công. Thay đổi này tiện lợi hơn nhưng mỗi lần tự động phản hồi đều tốn token tương đương một lần gửi prompt thông thường, có thể gây tăng chi phí không ngờ. Để tắt, thêm "respondToBashCommands": false vào settings.json. Tác giả khuyến nghị chỉ giữ true khi thực sự cần Claude phân tích output như khi test hay build.
