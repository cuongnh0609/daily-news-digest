B1Claude Code Auto Memory 運用の5つの落とし穴——CLAUDE.mdとの責任分界を正しく設計する

📰 本文 + 和越対訳

Claude Code の Auto Memory 機能を3ヶ月以上にわたって小規模チームで本番運用した経験をもとに、よくある5つの落とし穴と設計原則がZennで公開された。まず注意すべき点として、CLAUDE.md は人間が管理するプロジェクトルール（gitで版管理）であり、Auto Memory はClaude自身が自動記録する作業パターン（ローカル保存）という明確な役割分担がある。主な落とし穴として、MEMORY.mdが200行を超えると古い重要情報が読み込まれなくなること、CI/CDパイプラインでAuto Memoryが蓄積して再現不能なビルドが発生すること、旧来の設定情報が残って動作が予測不能になること、チームメンバー間でローカル保存のAuto Memoryに一貫性がなくなることなどが挙げられた。対策として、CI/CD環境ではCLAUDE_CODE_DISABLE_AUTO_MEMORY=1の設定を推奨。月次でメンテナンスし、100行近くなったらトピック別ファイルへの分割を行うべきとしている。

Một bài viết trên Zenn chia sẻ 5 cạm bẫy thường gặp và nguyên tắc thiết kế khi vận hành tính năng Auto Memory của Claude Code trong môi trường production suốt hơn 3 tháng. Điểm mấu chốt là phân biệt rõ CLAUDE.md (quy tắc dự án do con người viết, quản lý qua git) với Auto Memory (mẫu hành vi được Claude tự ghi lại, lưu trên máy local). Các bẫy chính gồm: MEMORY.md vượt 200 dòng sẽ mất thông tin cũ, CI/CD tích lũy Auto Memory gây build không reproducible, thông tin cũ gây hành vi không đoán trước, và thiếu nhất quán giữa các thành viên team. Khuyến nghị tắt Auto Memory trong CI/CD bằng biến môi trường CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 và bảo trì hàng tháng.

C34 LLM Agent Reliability Patterns from AI Engineer 2026: Verifier's Rule, Guardrails, Skill, Elicitation

📰 Article (English)

Jacob Lauritzen, CTO of legal-tech company Legora, presented at AI Engineer 2026 a framework of four complementary patterns for building reliable LLM agents. The core insight is that with LLM execution costs falling rapidly, planning and human review have become the new bottleneck — not task execution itself. Context Rot is identified as a fundamental problem: long chat-based agent interactions cause agents to "forget" initial instructions as conversation length grows, since chat forces tree/graph-shaped tasks into linear sequences. The four patterns: (1) Verifier's Rule — only assign tasks that have objectively verifiable correctness criteria; agents iterate until verification passes. (2) Guardrails — enforce least-privilege constraints at the tool/code level, never in prompts alone; destructive operations require explicit approval flows. (3) Skill — encode domain expertise as discrete DAG nodes (rule sets, few-shot examples, or API calls) so repeated human judgments become automated. (4) Elicitation — detect uncertainty via confidence thresholds and escalate to humans asynchronously; frequent escalation signals that logic should move into Skills. The framework emphasizes that patterns must be combined and layered, not used in isolation.
